# R Package Development Guide for Claude Code

A step-by-step playbook for building teaching-oriented R packages with Claude Code. Based on the process used to build the `wagnerwhitin` package for BUSML 4382 at Ohio State.

**Author:** Professor Vince Castillo
**Audience:** A Claude Code agent assisting with R package development

---

## Prerequisites

Before starting, ensure the following are available on the machine:

- **R** (version 4+)
- **RStudio** (optional but helpful for manual testing)
- **R packages:** `devtools`, `roxygen2`, `testthat`
- **Git** with push access to GitHub (use `gh auth setup-git` if using GitHub CLI with HTTPS)

---

## Phase 1: Planning

### 1.1 тАФ Write a PRD

Create a `PRD.md` in the repo root. It should define:

- **Purpose** тАФ what problem this package solves and who it's for
- **Function specifications** тАФ name, parameters, return values for every exported function
- **Output format** тАФ exact expected print output, plot descriptions
- **Test cases** тАФ at least 2 worked examples with known expected results
- **Design constraints** тАФ what dependencies are allowed, what to avoid
- **Package structure** тАФ file layout

### 1.2 тАФ Create an implementation plan

Before writing code, produce a detailed plan covering:

- File creation order (what depends on what)
- Algorithm pseudocode for non-trivial functions
- Naming conventions (be consistent across all functions)
- Parameter naming (decide once, apply everywhere тАФ e.g., if two functions share a concept, use the same parameter name in both)
- Test case expected values (compute by hand or verify independently)

**Lesson learned:** Parameter inconsistency between functions causes student confusion. Decide on a shared interface early and stick to it.

## Phase 2: Package Scaffolding

### 2.1 тАФ Directory structure

```
package-name/
тФЬтФАтФА DESCRIPTION
тФЬтФАтФА LICENSE
тФЬтФАтФА NAMESPACE
тФЬтФАтФА .Rbuildignore
тФЬтФАтФА .gitignore
тФЬтФАтФА R/
тФВ   тФЬтФАтФА main_function.R
тФВ   тФЬтФАтФА helpers.R
тФВ   тФФтФАтФА utils.R
тФЬтФАтФА man/                  # auto-generated by roxygen2
тФЬтФАтФА tests/
тФВ   тФЬтФАтФА testthat.R
тФВ   тФФтФАтФА testthat/
тФВ       тФЬтФАтФА test_main.R
тФВ       тФФтФАтФА test_helpers.R
тФЬтФАтФА README.md
тФФтФАтФА PRD.md
```

### 2.2 тАФ DESCRIPTION

```
Package: packagename
Type: Package
Title: One-Line Title
Version: 0.1.0
Authors@R: person("First", "Last", email = "email@osu.edu",
    role = c("aut", "cre"))
Description: Multi-line description of what the package does.
License: MIT + file LICENSE
Encoding: UTF-8
Imports: graphics, stats
Suggests:
    openxlsx,
    testthat (>= 3.0.0)
Config/testthat/edition: 3
RoxygenNote: 7.3.3
```

**Key points:**
- `RoxygenNote` must match the installed roxygen2 version. Check with `packageVersion("roxygen2")`.
- Use `Suggests` for optional dependencies (like `openxlsx` for Excel export). Use `Imports` for required ones.
- Only `graphics` and `stats` from base R need to be listed in Imports for R CMD check compliance.

### 2.3 тАФ LICENSE

For `License: MIT + file LICENSE` in DESCRIPTION, the LICENSE file must be DCF format:

```
YEAR: 2026
COPYRIGHT HOLDER: Your Name
```

**Not** the full MIT license text. R CMD check will throw a NOTE if this is wrong.

### 2.4 тАФ NAMESPACE

Write a placeholder тАФ roxygen2 will regenerate it:

```
# Generated by roxygen2: do not edit by hand
```

### 2.5 тАФ .Rbuildignore

Exclude non-package files from the build:

```
^\.claude$
^PRD\.md$
^LICENSE\.md$
^.*\.Rproj$
^\.Rproj\.user$
^AI_ASSISTANT_SYSTEM_PROMPT\.md$
^R_PACKAGE_DEV_GUIDE\.md$
```

Add any other `.md` docs, data files, or tooling configs that aren't part of the R package.

### 2.6 тАФ tests/testthat.R

Boilerplate test runner (never changes):

```r
library(testthat)
library(packagename)

test_check("packagename")
```

## Phase 3: Implementation

### 3.1 тАФ Write R source files

**Roxygen2 documentation pattern:**

```r
#' Function Title
#'
#' Description of what the function does.
#'
#' @param x numeric; description of parameter
#' @param y character; description of parameter
#'
#' @return Description of return value
#'
#' @examples
#' my_function(x = 1, y = "a")
#'
#' @export
my_function <- function(x, y) {
  # implementation
}
```

**Rules:**
- Every exported function needs `@export` and a complete roxygen block
- Internal helpers use `@noRd` to suppress documentation generation
- Use `@importFrom` for specific functions from other packages (e.g., `@importFrom stats qnorm sd`)
- For optional dependencies, use `requireNamespace("pkg", quietly = TRUE)` at runtime instead of `@importFrom`

**S3 methods pattern:**

```r
#' @export
print.MyClass <- function(x, ...) {
  # print logic
  invisible(x)  # always return invisible(x) from print methods
}

#' @export
plot.MyClass <- function(x, ...) {
  old_par <- par(no.readonly = TRUE)
  on.exit(par(old_par))  # always save/restore par
  # plot logic
}
```

### 3.2 тАФ Write tests

**Test file structure:**

```r
# ---------- Section Name ----------

test_that("description of what is tested", {
  result <- my_function(inputs)
  expect_equal(result$value, expected_value)
})
```

**What to test:**
1. **Input validation** тАФ bad types, NAs, negatives, wrong lengths
2. **Known examples** тАФ at least 2 worked examples with hand-verified expected values
3. **Edge cases** тАФ single element, zeros, boundary conditions
4. **Return structure** тАФ correct class, correct field names
5. **S3 methods** тАФ print returns `invisible(x)`, plot doesn't error
6. **Optional features** тАФ use `skip_if_not_installed("pkg")` for tests requiring suggested packages

### 3.3 тАФ Excel export (if needed)

A reusable pattern for exporting results to `.xlsx`:

```r
#' @export
export_xlsx <- function(x, file) {
  if (!inherits(x, "MyClass"))
    stop("x must be a MyClass object", call. = FALSE)
  if (!requireNamespace("openxlsx", quietly = TRUE))
    stop("Package 'openxlsx' is required. Install with: install.packages('openxlsx')",
         call. = FALSE)

  wb <- openxlsx::createWorkbook()
  openxlsx::addWorksheet(wb, "Sheet Name")
  openxlsx::writeData(wb, "Sheet Name", my_data_frame)
  openxlsx::saveWorkbook(wb, file, overwrite = TRUE)
  invisible(file)
}
```

## Phase 4: Build and Check

Run these in order after every round of changes:

```r
devtools::document()   # generate man/ pages and NAMESPACE from roxygen2
devtools::test()       # run testthat suite
devtools::check()      # full R CMD check
```

**Targets:**
- `devtools::test()` тАФ all expectations pass, 0 failures
- `devtools::check()` тАФ 0 errors, 0 warnings, 0 notes

**Common issues and fixes:**

| Issue | Fix |
|-------|-----|
| `Undocumented code objects` | Run `devtools::document()` to generate `.Rd` files |
| `License stub is invalid DCF` | Make LICENSE file DCF format (YEAR + COPYRIGHT HOLDER only) |
| `Non-standard file/directory found at top level` | Add the file to `.Rbuildignore` |
| `Hidden files and directories` | Add `.claude` (or similar) to `.Rbuildignore` |
| `RoxygenNote version mismatch` | Update `RoxygenNote` in DESCRIPTION to match installed version |

## Phase 5: Documentation

### 5.1 тАФ README.md

Structure for a teaching-oriented package:

1. **Title and one-line description**
2. **Installation** тАФ `devtools::install_github()` instructions
3. **Quick Start** тАФ full worked example with code and expected output
4. **Understanding the Output** тАФ section-by-section explanation of what the functions return
5. **How It Works** тАФ conceptual explanation of the algorithm/method (not just API docs)
6. **Parameter Notes** тАФ clarify any unit conventions, annualization, etc.
7. **Function Reference** тАФ table of all exported functions with one-line descriptions
8. **Sample Code** тАФ multiple examples covering all exported functions
9. **AI Assistant** тАФ link to the system prompt if one exists

### 5.2 тАФ AI Assistant System Prompt

Create an `AI_ASSISTANT_SYSTEM_PROMPT.md` to help students use the package. Structure:

1. **Identity and tone** тАФ who the assistant is, how it behaves
2. **Step-by-step onboarding flow** тАФ check RStudio тЖТ install devtools тЖТ install package тЖТ load тЖТ run example тЖТ explain output
3. **Output reference** тАФ exact expected output so the assistant can verify student results
4. **Behavior rules** тАФ one step at a time, wait for confirmation, never skip errors
5. **Troubleshooting** тАФ common errors and how to resolve them

Add to `.Rbuildignore` so it doesn't ship with the package.

## Phase 6: Git Workflow

### Commit pattern

Commit after each logical unit of work, not in giant batches:

```bash
git add <specific files>
git commit -m "Short description of change"
git push origin main
```

### Push setup (if using GitHub CLI)

If `git push` fails with credential errors:

```bash
gh auth setup-git          # configure gh as git credential helper
git push origin main       # now works via HTTPS
```

Or switch to SSH if keys are configured:

```bash
git remote set-url origin git@github.com:User/repo.git
```

## Checklist: Before Shipping

- [ ] All `devtools::test()` expectations pass
- [ ] `devtools::check()` returns 0 errors, 0 warnings, 0 notes
- [ ] README has installation instructions, quick start, and all function examples
- [ ] Every exported function has roxygen2 documentation with `@examples`
- [ ] Parameter names are consistent across all functions
- [ ] `.Rbuildignore` excludes all non-package files (PRD, guides, .claude, etc.)
- [ ] Manual test: install from GitHub with `devtools::install_github()`, restart R, run examples
- [ ] AI assistant system prompt is accurate and reflects final function signatures

## Iteration Lessons from wagnerwhitin

Things that came up during development that are worth knowing ahead of time:

1. **Decide parameter names once.** If multiple functions share a concept (e.g., a cost rate, a threshold, a time horizon), use the same parameter name in all of them. Refactoring parameter names after tests and docs are written is tedious and error-prone.

2. **Unit consistency matters for teaching.** If some functions need inputs in different units than others (e.g., monthly vs. annual), handle the conversion inside the function and document it clearly. Don't make students do the conversion themselves.

3. **Don't fight base R plotting.** Legend placement in single-panel plots was a recurring issue. Side-by-side panels (`par(mfrow = c(1, 2))`) with separate subtitles was simpler and cleaner than trying to overlay a legend.

4. **Excel export is high-value for students.** Adding `export_xlsx()` with `openxlsx` as a suggested dependency was straightforward and gives students a deliverable they can submit or paste into reports.

5. **LICENSE file format is picky.** `MIT + file LICENSE` in DESCRIPTION requires the LICENSE file to be DCF format (just YEAR and COPYRIGHT HOLDER), not the full MIT text.

6. **Always run `devtools::document()` before `devtools::check()`.** Missing `.Rd` files are the most common check warning.

7. **Test expected values by hand first.** Compute at least one example manually to verify the algorithm before writing tests. Tests that encode wrong expected values are worse than no tests.
